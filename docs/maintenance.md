# CopyShelf 维护与迭代手册

本文规定从需求进入到版本发布、用户验证和问题关闭的完整流程。它参考了外部项目的版本闭环，但以 CopyShelf 的 Windows-only、本地优先和隐私边界为准。

## 信息源

| 内容 | 唯一事实来源 |
| --- | --- |
| 产品方向、版本范围、交付状态 | `plan.md` |
| 外部 bug 和功能建议 | GitHub Issues |
| agent 内部调查任务 | `.scratch/`，规则见 `docs/agents/issue-tracker.md` |
| 架构与不可逆决策 | `docs/adr/` |
| 产品定位与竞品依据 | `docs/product/` |
| 用户可见能力和安装方式 | `README.md` / `README.en.md` |
| 发布结果 | Git tag、GitHub Actions、GitHub Release |

同一信息不在多个文件维护不同状态。版本范围变更先改 `plan.md`；架构边界变化必须增加 ADR，而不是只写在提交信息里。

## 工作优先级

每轮开始时按以下顺序检查，命中后先处理更高优先级事项：

1. **P0 安全与数据完整性**：数据丢失、凭据泄漏、片段误执行、不可恢复的同步覆盖、粘贴到错误窗口。
2. **P1 发布健康**：`main` 或 Release CI 失败、安装包不可用、应用无法启动、同步或全局快捷键普遍失效。
3. **P1 已验证用户问题**：有稳定复现步骤的高影响 Issue，进入 PATCH 版本。
4. **P2 可靠性回合**：每个 MINOR 功能版本之后至少安排一个 PATCH，处理反馈、测试稳定性和结构债务。
5. **P2 功能迭代**：从 `plan.md` 选择尚未完成的最高价值目标，进入 MINOR 版本。
6. **P3 探索**：只有完成前五项且有明确成功指标时才做实验。

不存在“为了凑版本而发布”。文档、CI 或内部重构若不改变用户行为，可以独立维护提交交付。

## 版本规则

遵循 SemVer：

- **PATCH**：bug 修复、可靠性、性能、兼容性和不改变产品承诺的内部重构。
- **MINOR**：向后兼容的新能力或明显扩展的工作流。
- **MAJOR**：数据格式、同步语义或产品边界的不兼容变化；发布前必须有 ADR 和迁移方案。

候选版本只允许一个主要结果，通常不超过三个紧密相关的用户故事。紧急修复从最近 Release 创建短分支，修复完成后合回 `main`，不直接在 `main` 上试错。

版本号必须同步：

- `pubspec.yaml`
- `lib/utils/constants.dart`
- `plan.md`
- Git tag `vX.Y.Z`

CI 会校验 tag 与 `pubspec.yaml` 的版本一致。

## 迭代状态机

`Proposed -> Ready -> In progress -> Review -> Windows validation -> Released -> Closed`

### Ready

进入开发前必须具备：

- 用户问题和预期结果，而不只是解决方案描述。
- 范围内与范围外内容。
- 可验证的验收条件。
- 数据、隐私、同步和 Win32 原生行为的风险判断。
- 需要更新的测试与文档清单。

### Definition of Done

一项工作只有同时满足以下条件才算完成：

- 验收条件逐项通过，错误与取消路径也有定义。
- 新行为有聚焦测试；修复先有失败复现或等价回归测试。
- `flutter analyze --no-fatal-infos` 和 `flutter test` 通过。
- 涉及 Windows 原生能力时，完成对应的实机检查；无法实测必须明确记录剩余风险。
- 不记录片段正文、剪贴板内容、Git 凭据或目标窗口敏感文本。
- 用户可见变化更新 README；版本进展更新 `plan.md`；架构决策更新 ADR。
- PR 只包含本轮相关文件，评审覆盖可靠性、易用性、性能、安全和回滚路径。
- 合并后 CI 通过；发布后安装包可下载、可启动，Issue 得到版本号与验证说明。

## 开发流程

1. **巡检**：查看开放 Issue、最近 CI、最新 Release 反馈和 `plan.md`。
2. **定版**：按优先级决定 PATCH 或 MINOR，写清主要结果、任务和验收条件。
3. **实现**：使用短分支和小提交；每个提交保持可理解、可测试，不混入无关格式化。
4. **评审**：先看行为回归、数据安全、Win32 生命周期和同步语义，再看代码风格。
5. **自动验证**：静态分析、全量测试、Windows Release 构建。
6. **实机验收**：只执行本次变更影响到的矩阵，并记录系统版本和结果。
7. **发布观察**：发布后验证安装、启动、升级和核心路径；至少观察 CI 与首轮反馈。
8. **闭环**：更新计划，回复并关闭已解决 Issue；未解决风险转成新 Issue，不藏在发布说明中。

## 验证分层

### 本地门禁

```bash
export PATH="$HOME/sdks/flutter-stable/bin:$PATH"
flutter analyze --no-fatal-infos
flutter test
git diff --check
```

静态分析的 warning 和 error 必须为零。现存 info 应逐步清理；在基线清零前不把 info 设为阻塞项。全仓格式化尚未形成稳定基线，因此当前只格式化新增文件和实际修改区域，禁止制造无关的大范围 diff。

### CI 门禁

- Pull Request：Windows 上静态分析、全量测试和 Release 构建。
- `main`：同样执行全部门禁，保证随时可发布。
- `v*` tag：校验版本、复用通过的门禁、构建 ZIP 并创建 GitHub Release。
- 同一分支的新提交会取消旧的未完成运行。

### Windows 实机矩阵

按变更影响选择，不要求每个 PATCH 重跑所有组合：

| 范围 | 最低验收 |
| --- | --- |
| 呼出与快捷键 | Windows 10/11、组合冲突、重绑失败恢复、重启后保持 |
| 窗口与粘贴 | 普通窗口、终端、多行确认、目标窗口关闭、多显示器/DPI |
| 输入 | 中文输入法、英文输入、Unicode、长文本、换行 |
| 系统边界 | 管理员窗口、RDP、休眠恢复、单实例、托盘重启 |
| Git 同步 | HTTPS/SSH、无凭据、超时、空仓库、分支不一致、冲突恢复 |
| 数据迁移 | 旧版本升级、导入重复项、导出再导入、损坏文件降级 |

管理员权限隔离可能阻止向高权限窗口注入输入，这是平台安全边界；产品必须明确降级和提示，不能用扩大应用权限掩盖。

## 发布与回滚

1. 确认 `plan.md` 中候选版本只剩实机与发布任务。
2. 同步版本号，更新中英文 README 的用户可见变化。
3. 合并 PR，等待 `main` 全部门禁通过。
4. 创建并推送带注释 tag：`git tag -a vX.Y.Z -m "vX.Y.Z"`。
5. 观察 Release workflow，下载 ZIP 做一次干净环境冒烟测试。
6. 将 `plan.md` 标记为已发布，并在对应 Issue 回复版本和 Release 地址。

若 Release 产物损坏，先撤下 Release 或标记为 prerelease，再修复并发布新的 PATCH；禁止移动或覆盖已经公开的 tag。若数据格式变更导致回滚风险，发布说明必须提供备份与降级步骤。

## Issue 闭环

Bug 至少包含版本、Windows 版本、复现步骤、预期、实际和发生频率。日志必须先脱敏；不要求用户公开片段正文、剪贴板、仓库地址或凭据。

关闭回复至少说明：

- 根因或行为变化。
- 首次包含修复的版本。
- Release 下载地址。
- 请用户验证的具体路径。

若无法复现，说明已尝试的环境和仍缺少的信息，保持 Issue 可继续补充；不要用“无法复现”代替调查结论。

## 例行维护

- **每轮**：Issue、CI、依赖安全提示、路线图状态。
- **每月**：Flutter 与直接依赖升级评估、GitHub Actions 版本、测试耗时与偶发失败、安装包体积。
- **每个 MINOR 后**：安排可靠性 PATCH，复查错误提示、取消路径、数据恢复和 Windows 兼容矩阵。
- **每季度**：核对竞品变化、成功指标和“明确不做”，删除低价值计划而不是无限扩张范围。

依赖升级独立提交，一次只升级一个相关集合；涉及 Win32、窗口管理、托盘或存储的升级必须跑对应实机路径。
